# Project Audit — Performance, Correctness, Consistency & Code Quality

**Date:** 2026-07-01
**Commit audited:** `58f3936` ("Fix Oracle SQL builder")
**Scope:** All projects under `src/` (core library + SqlServer, PostgreSql, Sqlite, MySql, Oracle, OpenTelemetry providers), plus build files. Tests and benchmarks were reviewed for context but not audited line-by-line.
**Method:** Static review. The audit environment has no .NET SDK, so no findings were validated by compiling/running; each finding cites the exact file and line so it can be verified quickly.

---

## Executive summary

The library is well structured (clean dialect/provider separation, cached metadata with compiled getters, LoggerMessage-based logging, activity tracing). The most impactful problems found:

| # | Severity | Area | Finding |
|---|----------|------|---------|
| 1 | High (perf) | All temp-table flows | `_bulk_insert_id` identity column is created on every temp table but **never used** — pure per-row overhead |
| 2 | High (perf) | SQLite | Default `BatchSize = 5` makes SQLite inserts ~2 orders of magnitude slower than they could be |
| 3 | High (bug) | PostgreSQL | Binary COPY writer is not disposed on failure — poisons the connection |
| 4 | High (bug) | Expression→SQL | String constants are not escaped and numeric/date constants use current culture — broken SQL / injection surface |
| 5 | Medium (bug) | Base + Oracle | Temp table is not dropped when the *insert into the temp table* fails; on Oracle the "temp" table is a **permanent** table, so it leaks |
| 6 | Medium (perf) | Return-entities flow | `FromSqlRaw` results are tracked by the change tracker; no `AsNoTracking()` |
| 7 | Medium (incons.) | SQLite merge | SQLite emits `ON CONFLICT` without a conflict target when `Match` is omitted (invalid SQL), while PostgreSQL/SQL Server/Oracle fall back to the primary key |
| 8 | Medium (bug) | MySQL/Oracle dialects | Base `Trim()` translation is PostgreSQL-specific `BTRIM`, inherited by MySQL and Oracle where it does not exist |
| 9 | Medium (bug) | Move/merge phase | `CopyTimeout` is never applied to the merge/`INSERT…SELECT` command — default 30 s ADO.NET timeout applies to the potentially longest phase |

Details below, grouped by category. File references are `path:line` at the audited commit.

## 1. Performance

### P1. `_bulk_insert_id` column is created but never used (High)

`BulkInsertProviderBase.CreateTableCopyAsync` always calls `AddBulkInsertIdColumn` (`src/PhenX.EntityFrameworkCore.BulkInsert/BulkInsertProviderBase.cs:188`), which runs:

- SQL Server: `ALTER TABLE #tmp ADD _bulk_insert_id INT IDENTITY PRIMARY KEY` (`SqlServerBulkInsertProvider.cs:17`)
- PostgreSQL: `ALTER TABLE tmp ADD COLUMN _bulk_insert_id SERIAL PRIMARY KEY` (`PostgreSqlBulkInsertProvider.cs:22`)
- MySQL: `ALTER TABLE tmp ADD _bulk_insert_id INT AUTO_INCREMENT PRIMARY KEY` (`MySqlBulkInsertProvider.cs:19`)

A repo-wide search shows the column is **never referenced again** — no `ORDER BY _bulk_insert_id`, no join, nothing. The cost is real: every temp-table row pays identity/sequence generation plus a PK b-tree insert (for large batches this is easily 10–30 % of temp-table write time), and one extra DDL round-trip per operation.

Presumably it was intended to guarantee that `ExecuteBulkInsertReturnEntities` returns rows in input order — but since `BuildMoveDataSql` has no `ORDER BY` (`Dialect/SqlDialectBuilder.cs:88-91`), that guarantee does not exist today anyway (see C7).

**Recommendation:** either drop the column entirely, or keep it and actually add `ORDER BY _bulk_insert_id` to the move/returning SQL. Either way, stop paying for it without using it.

### P2. SQLite default `BatchSize = 5` (High)

`SqliteBulkInsertProvider.CreateDefaultOptions()` sets `BatchSize = 5` (`SqliteBulkInsertProvider.cs:30-33`). Effective rows-per-INSERT is `Math.Min(options.BatchSize, MaxParams / columns.Count)` (`:139`), so with the default a 1 M-row insert executes **200 000 commands** instead of ~10 000 (10-column entity, `MaxParams = 1000`). This looks like a leftover debug value; even the XML doc table in `BulkInsertOptions` (`Options/BulkInsertOptions.cs:36-38`) advertises "SQLite: 5" as if it were a considered default.

Also note `MaxParams = 1000` (`SqliteBulkInsertProvider.cs:18`) matches SQLite < 3.32; every SQLite bundled with `Microsoft.Data.Sqlite` for years allows 32 766 parameters. Raising it would cut command count by another ~30×.

**Recommendation:** default `BatchSize` to something like `MaxParams / columnCount` (i.e. don't cap below the parameter limit) and consider raising `MaxParams`.

### P3. Returned entities are change-tracked (Medium)

`CopyFromTempTableAsync` returns `context.Set<TResult>().FromSqlRaw(query).AsAsyncEnumerable()` (`BulkInsertProviderBase.cs:233`). Every returned entity is attached to the change tracker: per-entity snapshotting, identity-map lookups, and memory growth linear in batch size — exactly what a bulk API is supposed to avoid. For a 500 k-row `ExecuteBulkInsertReturnEntitiesAsync` this is a large hidden cost, and a subsequent `SaveChanges()` may unexpectedly re-persist modifications to returned entities.

**Recommendation:** use `.AsNoTracking()` (optionally make tracking opt-in via `BulkInsertOptions`).

### P4. Converter chain is evaluated per value, even when irrelevant (Medium)

`ColumnMetadata.GetValue` loops over `options.Converters` for **every non-null value of every column of every row** (`Metadata/ColumnMetadata.cs:51-61`). SQL Server, PostgreSQL and MySQL providers install a geometry converter by default, so a plain 10-column × 1 M-row insert performs 10 M delegate calls + type checks that can never match for non-geometry columns.

**Recommendation:** resolve applicable converters once per column (e.g. ask the converter whether it can handle `ColumnMetadata.ClrType`, or pre-filter per column in the provider) and skip the loop for columns with no matching converter.

### P5. `EnumerableDataReader.IsDBNull` recomputes the value (Medium)

`IsDBNull(i)` is implemented as `GetValue(i) is DBNull` (`EnumerableDataReader.cs:77`). Bulk-copy implementations that call `IsDBNull` before `GetValue` cause the whole getter + converter chain to run **twice per field**. Caching the current row's values (or at least the last `(i, value)` pair) removes this.

Related contract nits in the same file: `GetOrdinal` returns `-1` instead of throwing for unknown names (`:53`), and `RecordsAffected` should be `-1` for SELECT-style readers, not `0` (`:59`).

### P6. `MetadataProvider` serializes all lookups behind one lock (Low–Medium)

`GetTableInfo<T>` takes a global `lock (_tablesPerContext)` on **every** call, including pure cache hits (`Metadata/MetadataProvider.cs:14`). Metadata construction (which compiles expression trees) also runs under the same lock, blocking all other threads. Under concurrent bulk inserts across many entity types this is measurable.

**Recommendation:** `ConcurrentDictionary<(Type, Type), TableMetadata>` with `GetOrAdd`, no lock.

### P7. Timeout not applied to the merge/move phase (Medium — perf-adjacent reliability)

`BulkInsertProviderBase.ExecuteAsync` creates commands without setting `CommandTimeout` (`BulkInsertProviderBase.cs:274-276`). `options.CopyTimeout` (default 10 min) is applied to the bulk copy itself, but the `INSERT INTO … SELECT` / `MERGE` that moves data out of the temp table — often the slowest step for conflict-handling inserts — runs with the provider's default timeout (typically 30 s). Large merges will spuriously time out.

**Recommendation:** `command.CommandTimeout = options.GetCopyTimeoutInSeconds()` in `ExecuteAsync` (needs `options` passed in; also applies to `DropTempTableAsync` implementations).

### P8. SQL Server `EnableStreaming` defaults to `false` (Low)

`SqlServerBulkInsertOptions.EnableStreaming = false` (`SqlServerBulkInsertOptions.cs:16`) makes `SqlBulkCopy` buffer the entire `IDataReader` result set in memory before sending. Since the source here is always a streaming reader over an `IEnumerable<T>`, `true` is the natural default for a bulk library (lower memory, earlier first-byte).

### P9. `DbContextOptionsExtensionInfo` allocated on every `Info` access (Low)

Both `BulkInsertOptionsExtension<TProvider>.Info` (`BulkInsertOptionsExtension.cs:14-15`) and `MetadataProviderExtension.Info` (`Metadata/MetadataProviderExtension.cs:8-9`) return `new …Info(this)` from the property getter. EF Core reads `Info` repeatedly (logging, service-provider caching, debug info); the standard pattern is to cache it in a field.

### P10. Minor allocations

- `TableMetadata.GetQuotedColumnName`/`GetColumnName` do a linear `FirstOrDefault` per lookup (`Metadata/TableMetadata.cs:75-89`); `GetUpdates` calls them per updated column → O(columns²) on wide tables. A `Dictionary<string, ColumnMetadata>` fixes it.
- `Helpers.AppendColumns` allocates a LINQ projection per call (`Helpers.cs:26-29`); it's called several times per statement build.
- `SqlServerGeometryConverter` rebuilds whole geometries with LINQ `Select(...).ToArray()` per value (`SqlServerGeometryConverter.cs:63-104`) — fine for occasional geometry, noticeable for geometry-heavy tables.

## 2. Correctness & reliability

### C1. PostgreSQL COPY writer leaks on failure (High)

`PostgreSqlBulkInsertProvider.BulkInsert` obtains the binary import writer with `var writer = …` and disposes it only on the success path at the end of the method (`PostgreSqlBulkInsertProvider.cs:53-121`). If a getter/converter throws, or the write itself fails mid-COPY, the writer is never disposed/cancelled; the `NpgsqlConnection` is left inside an unfinished `COPY … FROM STDIN`, and the subsequent rollback/close in `BulkInsert`'s `finally` operates on a broken connection.

**Recommendation:** wrap in `try/finally` (or `await using`) so the writer is always disposed — Npgsql's dispose cancels an incomplete COPY.

### C2. Expression-to-SQL constant rendering (High)

`SqlDialectBuilder.ToSqlExpression`, `ConstantExpression` branch (`Dialect/SqlDialectBuilder.cs:358-375`):

- **Strings are not escaped:** `return $"'{contantExpr.Value}'"` — any `'` in the constant breaks the statement; since this text is executed via `ExecuteNonQuery`/`FromSqlRaw`, it is also an injection surface if the constant ever originates from user data (e.g. `Where = (i, e) => i.Name == someUserString` — captured variables become constants after compiler closure evaluation only if written as literals; either way literals with quotes break).
- **Culture-sensitive formatting:** `contantExpr.Value?.ToString()` renders `double`/`decimal`/`DateTime` with the *current thread culture* — on a `fr-FR` server `3.14` becomes `3,14`, producing wrong or invalid SQL.

**Recommendation:** escape `'` → `''` for strings/Guids, and format numerics/dates with `CultureInfo.InvariantCulture` (dates ideally as ISO 8601 literals per dialect).

### C3. Temp table not dropped when the temp-table *insert* fails; Oracle leaks permanent tables (Medium)

In both `BulkInsertReturnEntities` and `BulkInsert` the flow is:

```csharp
var tableName = await PerformBulkInsertAsync(...); // creates temp table AND inserts into it
try { ...copy from temp... }
finally { await PerformDropTempTableAsync(...); }  // only covers the copy phase
```

(`BulkInsertProviderBase.cs:48-63` and `:102-110`.) If the insert into the temp table fails (constraint violation, converter exception, timeout), the drop never runs. For SQL Server/PostgreSQL/SQLite/MySQL the table is session-scoped so the damage is bounded, but **Oracle's "temp" table is a regular permanent table** (`OracleDialectBuilder.CreateTableCopySql` → `CreateTableCopySqlBase` → plain `CREATE TABLE`, `OracleDialectBuilder.cs:19-22`), so every failed conflict-handling insert leaves a `"#temp_bulk_insert_xxxxxxxx"` table behind.

**Recommendation:** move temp-table creation out of `PerformBulkInsertAsync` (or widen the `try`) so the `finally` covers the insert phase as well. Consider Oracle global temporary / private temporary tables (`ORA$PTT_…`) instead of permanent tables.

### C4. SQL Server MERGE is not concurrency-safe (Medium)

`SqlServerDialectBuilder.BuildMoveDataSql` emits `MERGE INTO target AS INSERTED` without `WITH (HOLDLOCK)`/`SERIALIZABLE` (`SqlServerDialectBuilder.cs:60`). The classic MERGE upsert race applies: two concurrent bulk upserts of the same key can both take the NOT MATCHED branch and one dies with a PK violation. Since this library's entire purpose is large concurrent-ish writes, `WITH (HOLDLOCK)` (or documenting the caveat) is warranted.

### C5. `SET IDENTITY_INSERT … ON` is triggered by *any* generated column (Medium)

`identityInsert` is `options.CopyGeneratedColumns && insertedColumns.Any(x => x.IsGenerated)` (`SqlServerDialectBuilder.cs:37`), but `ColumnMetadata.IsGenerated` is true for computed/default-valued columns too (`Metadata/ColumnMetadata.cs:24-28`). A table with a computed column (and no identity) would emit `SET IDENTITY_INSERT` — an error ("…is not a table with an identity column") — and conversely, computed columns cannot be inserted at all yet are included in `insertedColumns` when `CopyGeneratedColumns = true`. The `IsGenerated` predicate itself (four OR'ed conditions including a special case for non-Guid CLR types) deserves a comment and ideally a decomposition into `IsIdentity` / `HasDefault` / `IsComputed`.

### C6. Cancellation handlers that can never fire (Low–Medium)

SQL Server, MySQL and Oracle providers register a `RowsCopied` handler "to handle cancellation" when no progress options are set (`SqlServerBulkInsertProvider.cs:66-75`, `MySqlBulkInsertProvider.cs:84-93`, `OracleBulkInsertProvider.cs:96-105`), but in that branch `NotifyAfter` is left at its default of `0`, which **disables the event entirely** in all three drivers. The else-branches are dead code, and sync-path cancellation is in fact unsupported. Either set a `NotifyAfter` default in that branch or delete the dead handlers.

### C7. Return order of `ExecuteBulkInsertReturnEntities` is not guaranteed (Medium, documentation-level at minimum)

No dialect adds an `ORDER BY` when selecting from the temp table (`Dialect/SqlDialectBuilder.cs:88-91`, `SqlServerDialectBuilder.cs:129-133`). Users will naturally zip the returned list against their input to pick up generated keys; on SQL Server/PostgreSQL nothing guarantees that order once parallel plans or merges are involved. Fixing this is what the currently-unused `_bulk_insert_id` column is for (see P1). At minimum, document that order is unspecified.

### C8. Sync API does sync-over-async in the return-entities path (Medium)

The `sync` flag is threaded through the entire stack, but `CopyFromTempTableAsync` ignores it in the `returnData` branch — `FromSqlRaw(...).AsAsyncEnumerable()` is always async (`BulkInsertProviderBase.cs:230-234`) — and the public sync wrappers block on it with `.GetAwaiter().GetResult()` (`Extensions/PublicExtensions.cs:24-25`, `PublicExtensions.DbSet.cs:21-22`). That is textbook sync-over-async: thread-pool starvation risk and deadlock risk in synchronization-context environments. Either enumerate synchronously via `FromSqlRaw(...).AsEnumerable()` when `sync == true`, or drop the pretense and document the sync return-entities variants as blocking wrappers.

### C9. Empty-input handling is inconsistent and hostile (Low–Medium)

`BulkInsert`/`BulkInsertReturnEntities` throw `InvalidOperationException("No entities to insert.")` when `TryGetNonEnumeratedCount` reports 0 (`BulkInsertProviderUntyped.cs:39-42, 70-73`, duplicated again in `BulkInsertProviderBase.cs:83-86`). Two problems:

1. Throwing on an empty list is surprising — `AddRange`/`SaveChanges` treat empty as a no-op, and callers now need `if (list.Count > 0)` guards everywhere.
2. It only triggers for materialized collections. An empty *lazy* `IEnumerable<T>` (e.g. a LINQ filter that matches nothing) sails through and executes the whole temp-table machinery for zero rows — so the same logical input either throws or silently succeeds depending on collection type.

**Recommendation:** make empty input a no-op (return empty list) for all shapes; also delete the duplicated check in `BulkInsertProviderBase.BulkInsert`.

### C10. `GetConnection` can leak an opened connection (Low)

If `BeginTransaction` throws after the connection was opened, the newly opened connection is never closed (`Extensions/InternalExtensions.cs:34-72`). Rare, but a `try/catch` restoring the pre-call state would make ownership airtight. Related: `ConnectionInfo.WasBegan` semantics are inverted-sounding — `WasBegan == false` means "*we* began it" (`Metadata/ConnectionInfo.cs:7`); `OwnsTransaction` would read correctly.

### C11. Multiple provider registrations silently override each other (Low)

`BulkInsertOptionsExtension.ApplyServices` uses `AddScoped<IBulkInsertProvider, TProvider>()` (not `TryAdd`, `BulkInsertOptionsExtension.cs:20`). If a user chains `UseBulkInsertSqlServer().UseBulkInsertPostgreSql()` (e.g. copy-paste error), both register and the last one wins silently at resolution time. Detecting the double registration and throwing would fail fast.

### C12. Oracle dead/broken RETURNING branch (Low)

`OracleDialectBuilder.BuildMoveDataSql` emits `RETURNING … INTO :param` bind variables in the non-conflict path (`OracleDialectBuilder.cs:141-148`), but no parameters are ever bound and `OracleBulkInsertProvider.BulkInsertReturnEntities` throws `NotSupportedException` anyway (`OracleBulkInsertProvider.cs:35-45`). The branch is unreachable today and would fail if it ever became reachable. Delete it or implement it.

## 3. Provider inconsistencies

### I1. Conflict-target fallback to primary key: PostgreSQL yes, SQLite no (Medium — produces invalid SQL)

`PostgreSqlDialectBuilder.AppendConflictMatch` falls back to the PK when `Match` is null (`PostgreSqlDialectBuilder.cs:19-36`); SQL Server and Oracle do the same in their MERGE builders. SQLite uses the **base** `AppendConflictMatch`, which appends nothing when `Match` is null (`Dialect/SqlDialectBuilder.cs:155-164`) — and SQLite rejects `ON CONFLICT DO UPDATE` without a conflict target. So the same `OnConflictOptions { Update = … }` call works on four providers and fails with a syntax error on SQLite. The base implementation should adopt the PK-fallback (it is also what the DO-UPDATE grammar requires for SQLite).

### I2. `Trim()` translation defaults to PostgreSQL's `BTRIM` (Medium — invalid SQL on MySQL/Oracle)

Base: `Trim(lhs) => $"BTRIM({lhs})"` (`Dialect/SqlDialectBuilder.cs:153`). SQL Server and SQLite override to `TRIM`; PostgreSQL correctly uses `BTRIM`; **MySQL and Oracle do not override**, and neither has `BTRIM`. Update expressions support method calls on both, so `Update = (i, e) => new E { Name = e.Name.Trim() }` generates invalid SQL on MySQL/Oracle. The dialect-neutral default should be ANSI `TRIM`, with PostgreSQL overriding to `BTRIM` (or just use `TRIM` everywhere — PostgreSQL supports it too).

### I3. Transaction-type handling differs (Low)

SQL Server: `GetDbTransaction() as SqlTransaction` — a wrong transaction type silently becomes `null` and `SqlBulkCopy` runs with unexpected semantics (`SqlServerBulkInsertProvider.cs:40`). MySQL: explicit check that throws a clear error (`MySqlBulkInsertProvider.cs:56-61`). The MySQL approach is right; mirror it in SQL Server.

### I4. Column-mapping targets differ (Low)

SQL Server maps `PropertyName → ColumnName` (`SqlServerBulkInsertProvider.cs:79`), Oracle maps `PropertyName → QuotedColumName` (`OracleBulkInsertProvider.cs:109`), MySQL maps by ordinal (`MySqlBulkInsertProvider.cs:95-100`). Probably each is what its driver needs, but a comment explaining *why* Oracle needs the quoted form (case-sensitivity of quoted identifiers) would prevent "harmonizing" regressions.

### I5. Temp-table naming conventions drift (Cosmetic)

- MySQL temp table names start with `#` (`MySqlBulkInsertProvider.cs:22`) — a SQL Server convention with no meaning in MySQL (where `#` starts a comment when unquoted); it works only because the name is always backtick-quoted.
- SQLite drops the table-name component entirely (`SqliteBulkInsertProvider.cs:27`), Oracle drops it for length reasons (documented, good), others keep it. Harmless, but a shared helper with a per-provider max-length parameter would be tidier.

### I6. Oracle provider ignores `sync` and `CancellationToken` (Low)

`OracleBulkInsertProvider.BulkInsert` always calls blocking `WriteToServer` and returns `Task.CompletedTask` (`OracleBulkInsertProvider.cs:112-116`). ODP.NET has no async bulk copy, so blocking is unavoidable, but the async public API silently degrades to a thread-blocking call with no cancellation. Worth a doc note and a `ctk.ThrowIfCancellationRequested()` before starting.

### I7. MySQL bulk copy result/warnings ignored (Low)

`MySqlBulkCopy.WriteToServer(Async)` returns `MySqlBulkCopyResult` with `Warnings` (data truncation etc.) — currently discarded (`MySqlBulkInsertProvider.cs:104-112`). MySqlConnector docs recommend checking warnings because `LOAD DATA` can silently coerce/truncate values. Consider surfacing them via logging or an option to throw.

## 4. API design

### A1. 24 near-identical public overloads

`PublicExtensions.DbContext.cs` + `PublicExtensions.DbSet.cs` define 4 operations × {DbContext, DbSet} × {provider-options, common-options, no-options} = 24 methods that all funnel into two core methods. The no-options overloads could be collapsed into the options ones with `Action<TOptions>? configure = null` defaults, halving the surface. Not urgent (breaking-change considerations), but worth planning for the next major version.

### A2. `dbContext.Set<T>() ?? throw` is dead code

`DbContext.Set<T>()` never returns null (`PublicExtensions.DbContext.cs:21-22, 68-69, 116-117, 162-163, 209-210`). The null-coalescing throw is misleading; if the entity type isn't in the model the failure happens later in `GetTableInfo` with a better message anyway.

### A3. Redundant context resolution

`ExecuteBulkInsertReturnEntitiesCoreAsync` and its siblings call `InitProvider(dbSet, …)` (which resolves the context) and then call `dbSet.GetDbContext()` **again** for `GetTableInfo` in the same expression (`PublicExtensions.cs:25-27`, `PublicExtensions.DbSet.cs:110-113, 158-161, 205-208`). Use the `context` already returned by `InitProvider`.

### A4. `CopyFromTempTableAsync<T, TResult>` has a pointless second type parameter

`TResult` is always `T` at both call sites (`BulkInsertProviderBase.cs:52, 105`).

### A5. Duplicate `RawWhere`/`Where` mutual-exclusion validation

The "Cannot specify both RawWhere and Where" check is duplicated in three dialects (`Dialect/SqlDialectBuilder.cs:112-117`, `SqlServerDialectBuilder.cs:83-88`, `OracleDialectBuilder.cs:112-117`) and only fires when there are updates. Validate once, up front, when the options object is consumed.

They're the same object today; the duplication invites a future mismatch. Use the parameter.

Oracle uses `""` for the same purpose. Pick one mechanism (the empty-string check in the base already exists).

## 6. Build, packaging & docs

1. **Package description is outdated:** `Directory.Build.props` says "…on SQL Server, PostgreSQL and SQLite" — MySQL and Oracle are missing (this text ships on nuget.org for every package).
2. `<Copyright>Fabien Ménager © 2025</Copyright>` — stale year.
3. `<LanguageVersion>12</LanguageVersion>` pinned while targeting net10.0 — intentional lowest-common-denominator? A comment would help; otherwise consider `latest` per-TFM.
4. The commented-out `<AnalysisMode>AllEnabledByDefault</AnalysisMode>` suggests analyzers were tried and abandoned; several findings above (CA1310-style culture issues, dead code) would have been caught. Consider re-enabling incrementally with a baseline.
5. `README`/docs: worth documenting explicitly per provider: return-order non-guarantee (C7), Oracle sync-only + no cancellation (I6), MySQL `AllowLoadLocalInfile=true` connection-string requirement (currently surfaces as an obscure driver error), and MERGE concurrency caveat (C4).

## Suggested priority order

**Quick wins (small diffs, high value):**
1. Remove (or actually use) `_bulk_insert_id` (P1/C7).
2. Raise SQLite default batch size (P2).
3. `try/finally` around the PostgreSQL COPY writer (C1).
4. Escape + invariant-culture constants in `ToSqlExpression` (C2).
5. Base `Trim` → `TRIM`, PostgreSQL override → `BTRIM` (I2).
6. PK fallback for SQLite conflict target (I1).
7. Apply `CopyTimeout` in `ExecuteAsync` (P7).
8. Fix the NuGet description (§6.1).

**Medium efforts:**
- Widen temp-table cleanup scope; Oracle temp-table strategy (C3).
- Per-column converter resolution (P4) and `IsDBNull` caching (P5).
- `AsNoTracking` (or opt-in tracking) for returned entities (P3).
- Empty-input no-op semantics (C9).
- `MERGE … WITH (HOLDLOCK)` on SQL Server (C4).

**Next major version:**
- Consolidate the 24 public overloads (A1).
- Rename typo'd public-ish identifiers (`QuotedColumName`, `WasBegan`, `MySqlServerDialectBuilder`) (§5).
